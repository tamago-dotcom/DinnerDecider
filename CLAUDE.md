# DinnerDecider プロジェクト

## プロジェクト概要
今夜なに食べる？アプリ。
35問の質問バンクからランダム5問を出題し、回答をスコア化して7カテゴリ（ストレス・疲労・緊張・活気・社会性・欲求・混乱）の合計点を算出。決定木で10グループに分類し、夕食を提案する。フィードバックデータを蓄積してscikit-learnで再学習することで精度を上げていく。

## 技術スタック
- バックエンド: Python + Flask
- DB: SQLite (dinner.db)
- フロントエンド: HTML + CSS + JavaScript
- ML: scikit-learn（決定木）
- 起動: python app.py → http://localhost:5000

## ブランチ運用
- main: 本番（動く状態を常に保つ）
- develop: 開発用（ここで作業してmainにマージ）

## エージェント構成

### プロジェクトマネジメントエージェント（PM）
ユーザーの指示を受け取り、以下の各エージェントに作業を割り振る。
全エージェントの作業完了後、結果をまとめてユーザーに報告する。
指示の受け取り方：「〜〜したい」という指示を受けたら各エージェントに割り振り、完了後に報告。

### 市場調査エージェント
- ユーザーのニーズ分析
- 競合アプリの調査
- 機能の優先順位提案
- UX改善提案

### コーディングエージェント
- 機能実装（Flask・HTML・CSS・JS・Python・scikit-learn）
- 既存コードとの整合性を保った修正
- developブランチで作業
- 実装後は必ずgit add・git commitまで行う

### 単体テストエージェント
- 各関数・モジュール単位の動作確認
- エッジケースのチェック
- Pythonコードの構文チェック
- テスト結果をTEST_REPORT.mdに記録

### 結合テストエージェント
- 画面遷移の動作確認
- APIエンドポイントの動作確認
- DBへの保存・取得の確認
- フロントエンドとバックエンドの連携確認
- テスト結果をTEST_REPORT.mdに追記

### 仕様書作成エージェント
- 実装内容をSPEC.mdに反映
- 変更履歴を記録
- APIエンドポイント一覧を更新

## ファイル構成
- app.py: Flaskメイン・ルーティング
- database.py: SQLite操作
- ml_model.py: 決定木・再学習ロジック
- questions_data.py: 35問の質問バンク
- templates/: HTMLテンプレート
- static/: CSS・JS
- SPEC.md: 仕様書
- TEST_REPORT.md: テスト結果
- CLAUDE.md: このファイル

## 実装予定・進行中の機能
### フィードバック機能（実装中）
- 結果画面に「実際に何を食べましたか？」の入力フォームを追加
- 提案4品からの選択ボタン＋「その他」自由入力
- 満足度入力（1〜5の星評価）
- feedbackテーブルをSQLiteに追加（id・session_id・actual_food・satisfaction・created_at）
- 診断結果の7カテゴリスコアと紐づけて保存
- フィードバック50件以上で再学習ボタンを表示
- 再学習時は「実際に食べた料理×7カテゴリスコア」でscikit-learnが学習

## 開発時の必須ルール

### 全ての修正・実装時に必ず行うこと
1. コーディングエージェントが実装・修正
2. 単体テストエージェントが各関数・モジュールの動作確認を実施
3. 結合テストエージェントが画面遷移・API・DB連携の動作確認を実施
4. テスト結果を必ずTEST_REPORT.mdに日時付きで追記
5. 仕様書エージェントがSPEC.mdを更新
6. git commitしてdevelopにpush
7. テスト全通過後にmainにマージしてpush

### TEST_REPORT.mdの記録フォーマット
## [日時] [修正内容のタイトル]
### 単体テスト
- [テスト項目]: ✅ or ❌
### 結合テスト
- [テスト項目]: ✅ or ❌
### 結果: 全通過 or 要修正

### 注意事項
- テストが1つでも❌の場合はmainにマージしない
- 必ずdevelopブランチで作業してからmainにマージする
