# DinnerDecider 仕様書

## バージョン
- v1.0: 初期リリース（診断機能・履歴機能）
- v1.1: フィードバック機能追加（2026-03-20）

## システム概要

「今夜なに食べる？」をテーマにした夕食提案アプリ。35問の質問バンクからランダムに5問を出題し、回答をスコア化してストレス・疲労・緊張・活気・社会性・欲求・混乱の7カテゴリ得点を算出する。scikit-learnの決定木モデルで10グループに分類し、その日の気分に合った夕食を提案する。ユーザーのフィードバックデータを蓄積して再学習することで提案精度を向上させる。

- バックエンド: Python + Flask
- DB: SQLite（dinner.db）
- フロントエンド: HTML + CSS + JavaScript
- ML: scikit-learn（決定木 DecisionTreeClassifier）
- 起動: `python app.py` → http://localhost:5000

## 機能一覧

### v1.0 機能

#### 診断機能
- 35問の質問バンク（`questions_data.py`）から7カテゴリ各1問、計5問をランダムに出題
- 各回答値（0〜3）をカテゴリスコアとして集計
- 決定木モデルでグループ（G1〜G10）を判定し、対応する夕食4品を提案
- G10「普通の日」のみランダムで1品ハイライト表示
- 診断結果をレーダーチャートと棒グラフで可視化
- 結果はDBに保存され、URLで再参照可能（`/result/<result_id>`）

#### 履歴機能
- 過去の診断結果一覧をページネーション（10件/ページ）で表示（`/history`）
- フィードバック件数の表示
- フィードバック50件以上で再学習ボタンを表示

### v1.1 機能

#### フィードバック機能
- 診断結果画面で「実際に何を食べたか」と「満足度（1〜5）」を収集
- 提案4品から選択するボタンUI＋「その他」自由入力
- 同一診断結果への再送信は上書き保存（INSERT or UPDATE）
- フィードバックデータをMLモデルの再学習に活用

## フィードバック機能 詳細仕様

### UI仕様

`templates/result.html` および `static/js/feedback.js` で実装。

- **提案4品の選択ボタン**: 診断結果の提案料理4品がそれぞれボタンとして表示される
- **「その他」ボタン**: 提案外の料理を入力するためのボタン。選択するとテキスト入力欄（`#feedbackOtherInput`）が表示される
- **星評価（1〜5）**: `feedback.js` がDOMに星ボタン（`★`）を1〜5で動的生成。ホバーでプレビュー表示、クリックで確定
- **送信ボタン**: 料理の選択（その他の場合は入力必須）と星評価の両方が揃った場合のみ活性化（`disabled` 解除）
- **送信後の完了メッセージ**: フォームが非表示になり「ありがとうございました！🙏 あなたのフィードバックでAIが賢くなります」というメッセージを表示

### バリデーション仕様

**クライアントサイド（feedback.js）**
- 料理ボタンまたは「その他」テキスト入力が空でない場合のみ送信ボタン活性化
- 星評価が1以上の場合のみ送信ボタン活性化

**サーバーサイド（app.py: `POST /api/feedback`）**
- `actual_food`: 必須。空文字・空白のみ不可（`.strip()` 後に空ならエラー）
- `satisfaction`: 必須。1〜5の整数のみ受付（`isinstance(satisfaction, int)` かつ `1 <= satisfaction <= 5`）
- `result_id`: `quiz_results` テーブルに存在すること（`db.get_result(result_id)` で確認）
- 同一 `result_id` への重複送信: 既存レコードを上書き（UPDATE）。存在しない場合は新規登録（INSERT）

### ML再学習への組み込み

`ml_model.py` の `retrain()` メソッドおよび `food_to_group()` 関数で実装。

- **food_to_group()**: 料理名から対応するグループIDを検索する関数。`GROUP_INFO` の `foods` リストを走査し、完全一致または前方一致でグループIDを返す。マッチしない場合は `None` を返す
- **満足度4以上の重みづけ**: `satisfaction >= 4` のフィードバックは同じデータを2回追加することで学習の重みを増加させる（高満足度データを優遇）
- **フリーテキスト（未登録料理）のMLスキップ**: `food_to_group()` が `None` を返した場合、そのフィードバックはML訓練データに含めない
- **合成データとの統合**: 再学習時は初期合成データ（各グループ120件）＋実ユーザーデータ＋フィードバックデータを結合して学習

### 再学習トリガー

- **条件**: `quiz_results` テーブルのデータが50件以上であること（フィードバック件数ではなく診断件数）
- **場所**: 履歴画面（`/history`）に再学習ボタンが表示される（`feedback_count` が渡され表示制御に使用）
- **エンドポイント**: `POST /api/retrain`
- **モデル保存**: 再学習後のモデルは `model.pkl` に上書き保存される
- **レスポンス例（成功時）**: `{"success": true, "message": "N 件のデータで再学習しました（精度: XX.X%）"}`
- **レスポンス例（データ不足時）**: `{"success": false, "message": "再学習にはデータが50件必要です（現在 N 件）"}`

## 既知の課題・今後の対応

- **料理名の重複によるMLマッピングへの影響**: `food_to_group()` は先頭からの前方一致・完全一致で最初にマッチしたグループIDを返す仕様のため、「焼肉」はG1（限界戦士）とG2（ご褒美モード）の両方に存在し、G1が優先されてしまう。同様に「鍋」はG3（癒しが欲しい）・G7（社交モード）の両方に存在し、G3が優先される
- **フィードバック50件未満では再学習不可**: 診断件数が50件未満の場合は再学習APIがエラーを返す
- **今後の実装予定**: Renderでの公開
